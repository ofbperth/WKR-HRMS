# Cloud Data Management Security Phase

## Architecture

Browser traffic must use HTTPS/TLS and call only Next.js API routes or server actions. Sensitive RM data is encrypted in the backend before database writes, then stored in Supabase PostgreSQL. Object attachments must use a private Supabase Storage bucket and signed download URLs only.

## Sensitive Fields

Only these fields are treated as sensitive in this phase:

- HN
- AN
- RCA narrative
- Reporter name

Analytics fields such as severity, risk code, unit, status, category, sentinel flag, and dates remain plaintext so dashboards, heatmaps, trend reports, filtering, search, and exports do not depend on decrypted data.

## Encryption

The reusable backend utility is in `lib/encryption.ts`:

- `encrypt()`
- `decrypt()`

It uses AES-256-GCM with a random 12-byte IV and stores JSON containing `ciphertext`, `iv`, and `authTag`.

Required environment variable:

```bash
ENCRYPTION_KEY="<32 random bytes, base64 or hex>"
```

Generate a base64 key:

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
```

The key must never be committed or exposed through `NEXT_PUBLIC_*`.

## Migration and Backfill

The additive Prisma migration adds:

- `Incident.hn_encrypted`
- `Incident.an_encrypted`
- `Incident.reporter_name_encrypted`
- `RCA.rca_encrypted`
- `AuditLog.role`
- `AuditLog.ip_address`

Existing plaintext columns are preserved for compatibility. Run the backfill after setting `ENCRYPTION_KEY`:

```bash
npm run security:backfill-sensitive
```

The backfill encrypts existing plaintext HN, AN, reporter name, and RCA narrative values, validates HN/AN decryption, and does not delete legacy plaintext.

## Supabase Hardening

Apply `prisma/supabase_rls_storage.sql` in production after Prisma migrations. It:

- Enables and forces RLS on all app tables.
- Revokes direct `anon` and `authenticated` table access.
- Allows backend `service_role` access only.
- Makes `AuditLog` append-only with update/delete blocking triggers.
- Creates a private `risk-attachments-private` bucket.
- Restricts storage access to backend service-role operations.

The application must continue enforcing role scope in Next.js:

- Reporter: own reports only.
- Unit Manager: own unit only.
- RM/Admin: existing RM/Admin scope.

Do not expose service role credentials to the browser.

## Attachments

No public object URLs are allowed. Attachment downloads must be generated by a backend route with signed URLs expiring in 1 hour or less. Keep the bucket private.

## Deployment Checklist

- Configure HTTPS/TLS at the hosting layer.
- Set secure production secrets: `DATABASE_URL`, `AUTH_SECRET`, `ENCRYPTION_KEY`.
- Ensure cookies are `httpOnly`, `sameSite=lax`, and `secure` in production.
- Run Prisma migrations.
- Run `npm run security:backfill-sensitive`.
- Apply `prisma/supabase_rls_storage.sql`.
- Confirm no `NEXT_PUBLIC_*` secret contains database, service role, or encryption keys.
- Verify dashboards, heatmap, analytics, RCA workflow, reports, search/filter, exports, auth, and RBAC.
- Verify sensitive reveal/decrypt actions are audit logged.
